## Excluding API Routes Using Negative Lookahead

The matcher config allows full regex, including negative lookaheads to match all except specific paths. The most common pattern for excluding API routes is:

```typescript
export const config = {
  matcher: [
    '/((?!api|_next/static|_next/image|favicon.ico).*)',
  ],
}
```

This is a negative lookahead regex that excludes paths starting with /api, /login, and Next.js internal paths.

## Alternative Pattern Examples

Another variation uses: `matcher: ["/", "/((?!studio|api).{1,})"]` to not match any route with studio or api in it.

For more specific exclusions, you can add multiple paths:

```typescript
export const config = {
  matcher: [
    '/((?!api|login|_next/static|_next/image).*)',
  ],
}
```

## Using Conditional Logic Instead

If you cannot do this with matcher because it only accepts simple path patterns, you'll need to use conditional statement:

```typescript
export function middleware(request: NextRequest) {
  if (request.nextUrl.pathname.startsWith('/api/')) {
    return NextResponse.next()
  }
  // your middleware logic
}
```

The matcher values need to be constants so they can be statically analyzed at build-time.

Both approaches are valid in 2024, with the matcher pattern being more declarative and the conditional logic providing more flexibility for complex scenarios.

### For Route Handlers (including PATCH)

When working with PATCH requests in dynamic routes, the proper syntax is:

The params is a promise that resolves to an object containing the dynamic route parameters for the current route, and you need to await it like: `export async function GET(request: Request, { params }: { params: Promise<{ slug: string }> }) { const { slug } = await params }`
